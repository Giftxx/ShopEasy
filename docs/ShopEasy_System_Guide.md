## 1. ภาพรวมระบบทั้งหมด

ShopEasy คือแพลตฟอร์ม E-Commerce ที่มี **AI ช่วยดูแลลูกค้าอัตโนมัติ** ตั้งแต่การติดตามพัสดุไปจนถึงการอนุมัติการคืนเงิน โดยมีทีม Admin และ AI Engineer ช่วยดูแลระบบอีกชั้นหนึ่ง

### สรุปในหนึ่งประโยค
> ลูกค้าถาม → AI เข้าใจ → AI ค้นหาข้อมูลในระบบ → AI ตอบ → บันทึกทุกขั้นตอนไว้ตรวจสอบ

### ใครใช้ระบบนี้บ้าง

| กลุ่มผู้ใช้ | บทบาท | เข้าถึงผ่านหน้าจอไหน |
|-----------|------|-------------------|
| ลูกค้า (Customer) | ถามเรื่องออเดอร์, ขอคืนเงิน, ดูสถานะพัสดุ | Customer Portal |
| แอดมิน (Admin) | อนุมัติการคืนเงิน, ดูเคส, จัดการลูกค้า | Admin Portal |
| AI Engineer | ดู trace ของ AI, ทดสอบ workflow, ดู metrics | AI Control Portal |

## 2. สามหน้าจอหลัก (Portals)

### Customer Portal — หน้าจอของลูกค้า
ลูกค้าเข้ามาที่หน้าจอนี้ได้ 5 สิ่ง:

- **แชทกับ AI** — พิมพ์คำถามใดก็ได้ AI จะตอบโดยอิงจากข้อมูลออเดอร์จริงในระบบ
- **คำสั่งซื้อของฉัน** — ดูรายการออเดอร์ทั้งหมด กรองตามสถานะได้
- **การจัดส่งของฉัน** — ดูสถานะพัสดุแต่ละชิ้น พร้อม timeline ว่าพัสดุผ่านที่ไหนมาบ้าง
- **คืนเงิน / คืนสินค้า** — ยื่นคำขอคืนเงิน แนบหลักฐาน ติดตามสถานะ
- **การแจ้งเตือน** — รับรู้ก่อนว่ามีปัญหากับพัสดุ ก่อนที่จะต้องโทรถาม

### Admin Portal — หน้าจอของทีม Operations
แอดมินดูแลงานผ่าน 4 ส่วนหลัก:

- **Dashboard** — ตัวเลขสรุปภาพรวม: ออเดอร์วันนี้, การอนุมัติที่รอ, เคสเปิดอยู่
- **Pending Approvals** — รายการที่ AI เสนอให้อนุมัติ เช่น การคืนเงินความเสี่ยงสูง โดย AI จะอธิบายเหตุผลไว้ให้แล้ว
- **Cases** — เคสสนับสนุนลูกค้าทั้งหมด สร้างได้ทั้งจาก AI อัตโนมัติ และจาก Admin เอง
- **Customers** — ข้อมูลลูกค้า, ออเดอร์, ประวัติ

### AI Control Portal — หน้าจอของ AI Engineer
ใช้สำหรับ Monitor และทดสอบระบบ AI:

- **Workspace** — กด Run เพื่อทดสอบ workflow ด้วยข้อมูลจริง ดูผลลัพธ์ทันที
- **Traces** — ดู log ของทุก AI run ว่าผ่าน node ไหน ใช้เวลาเท่าไหร่ ตัดสินใจอะไร
- **Tool Logs** — ดูว่า AI เรียก tool อะไรบ้าง ส่งอะไรไป ได้ผลอะไรกลับมา
- **Metrics** — กราฟแสดง performance: จำนวน workflow ที่รัน, อัตราสำเร็จ, เวลาตอบสนอง

## 3. เส้นทางของข้อมูล — จากลูกค้าถึงฐานข้อมูล

เมื่อลูกค้าพิมพ์ข้อความ ข้อมูลวิ่งผ่านระบบแบบนี้:

```
ลูกค้าพิมพ์: "ของฉันอยู่ไหนแล้ว"
        ↓
[Frontend — React] รับข้อความ แสดงผล
        ↓
[Backend API — FastAPI] รับคำขอ ตรวจสอบสิทธิ์ว่าใครส่งมา
        ↓
[Chat Service] เลือกว่าจะส่งต่อไป Workflow ไหน
        ↓
[AI Router] ให้ LLM วิเคราะห์ว่าคำถามนี้เป็นเรื่อง "ติดตามพัสดุ"
        ↓
[Workflow 01 — Tracking] ดึงข้อมูล → วิเคราะห์ → ตอบ
        ↓
[Database] บันทึกการสนทนา, trace, tool logs
        ↓
[Frontend] แสดงคำตอบให้ลูกค้า
```

ทั้งหมดนี้ใช้เวลาประมาณ **2–10 วินาที** ขึ้นอยู่กับความซับซ้อนของคำถาม

## 4. หัวใจของระบบ — AI Agent คืออะไร

### Agent ≠ Chatbot ธรรมดา

Chatbot ธรรมดา: ลูกค้าถาม → Bot ตอบจาก script ที่เขียนไว้

AI Agent ของ ShopEasy: ลูกค้าถาม → AI **คิด** → AI **ค้นหาข้อมูล** → AI **ตัดสินใจ** → AI **ตอบ**

### Node คืออะไร

ให้นึกภาพว่า Agent ทำงานผ่าน "สถานี" หลายๆ สถานีต่อกัน แต่ละสถานีทำหน้าที่เฉพาะอย่าง:

```
[สถานี 1: Router]     → ฉันเข้าใจว่าลูกค้าถามเรื่องอะไร
[สถานี 2: Memory]     → ฉันเคยคุยกับลูกค้าคนนี้เรื่องอะไรมาก่อน?
[สถานี 3: Context]    → ลูกค้าคนนี้มีออเดอร์ไหนที่เกี่ยวข้อง?
[สถานี 4: Planner]    → ฉันควรดึงข้อมูลอะไรบ้าง?
[สถานี 5: Shipping]   → สถานะพัสดุตอนนี้เป็นยังไง?
[สถานี 6: Respond]    → สรุปและตอบลูกค้าเป็นภาษาธรรมชาติ
[สถานี 7: Memory]     → จำไว้ว่าเพิ่งคุยเรื่องอะไร
```

ข้อมูลไหลจากสถานีหนึ่งไปอีกสถานีอัตโนมัติ เรียกว่า **LangGraph**

### Tool คืออะไร

เมื่อ Agent อยู่ที่สถานีไหนสถานีหนึ่ง มันอาจต้อง "หยิบเครื่องมือ" มาใช้ เช่น:

| Tool | ทำอะไร |
|------|-------|
| get_customer_orders | ดึงรายการออเดอร์ของลูกค้าจากฐานข้อมูล |
| get_shipment_status | เช็คสถานะพัสดุล่าสุด |
| search_policy | ค้นหานโยบายที่เกี่ยวข้องกับคำถาม |
| calculate_risk_score | คำนวณว่าการคืนเงินนี้มีความเสี่ยงแค่ไหน |
| create_case | สร้างเคสสนับสนุนลูกค้าโดยอัตโนมัติ |

### ReAct Loop — AI ที่คิดซ้ำได้

ระบบมี "ReAct Engine" ซึ่งทำงานแบบนี้:

```
คิด (Reason) → ลองทำ (Act) → ดูผล (Observe) → คิดอีกครั้ง → ลองทำอีกครั้ง
```

ถ้า tool แรกที่ใช้ไม่ได้ผลลัพธ์ที่ต้องการ AI จะลองวิธีอื่น แทนที่จะยอมแพ้ทันที

## 5. Workflow 01 — ติดตามพัสดุ

**เมื่อไหร่:** ลูกค้าถามว่าออเดอร์หรือพัสดุอยู่ไหน

**ลำดับการทำงาน:**

1. **รับข้อความ** — ลูกค้าพิมพ์ว่า "ของฉันอยู่ไหนแล้ว"
2. **จำแนกเจตนา** — AI รู้ว่าคำถามนี้เกี่ยวกับการติดตาม ไม่ใช่เรื่องคืนเงิน
3. **ดึงความจำ** — เคยคุยกับลูกค้าคนนี้เรื่องออเดอร์ไหนมาก่อนไหม?
4. **หาบริบท** — ลูกค้ามีออเดอร์ไหนที่ยังเปิดอยู่บ้าง?
5. **วางแผน** — ควรดูข้อมูลอะไรก่อน?
6. **ดึงสถานะพัสดุ** — ดึงจาก shipping system ว่าพัสดุแต่ละชิ้นอยู่ที่ไหน
7. **ตอบลูกค้า** — สรุปเป็นภาษาไทยเข้าใจง่าย
8. **บันทึกความจำ** — จดไว้ว่าเพิ่งคุยเรื่อง ORD-1001
9. **บันทึก log** — บันทึกทุกขั้นตอนเพื่อให้ทีม AI ตรวจสอบได้

**สิ่งที่ลูกค้าได้รับ:**
> "พัสดุของคุณ (SHP-9001) อยู่กับ Flash Express ระหว่างทาง คาดว่าถึงพรุ่งนี้ก่อน 17.00 น. ค่ะ หากยังไม่ถึงภายในวันที่ XX กรุณาแจ้งเราอีกครั้ง"

## 6. Workflow 02 — ขอคืนเงิน

**เมื่อไหร่:** ลูกค้าต้องการคืนสินค้าหรือขอเงินคืน

**ขั้นตอนการทำงานแบบละเอียด:**

```
ขั้น 1: ลูกค้าแจ้งว่าต้องการคืนเงิน พร้อมระบุเหตุผล
          ↓
ขั้น 2: AI ค้นหาออเดอร์ที่เกี่ยวข้องในระบบ
          ↓
ขั้น 3: AI ตรวจสอบนโยบายการคืนเงิน:
          - อยู่ในช่วงเวลาที่รับคืนได้ไหม?
          - เหตุผลตรงกับเงื่อนไขไหม?
          - สินค้าประเภทนี้คืนได้ไหม?
          ↓
ขั้น 4: AI คำนวณ Risk Score (คะแนนความเสี่ยง):
          - 0–69: ความเสี่ยงต่ำ → AI อนุมัติเองได้
          - 70–89: ความเสี่ยงกลาง → Admin ต้อง approve
          - 90–100: ความเสี่ยงสูงมาก → ต้องมีเอกสารประกอบ
          ↓
ขั้น 5: สร้างเคสและ record ในระบบ
          ↓
ขั้น 6 (ถ้า Risk ≥ 70): ส่งไปที่ Admin ให้พิจารณา
ขั้น 6 (ถ้า Risk < 70):  อนุมัติอัตโนมัติ แจ้งลูกค้าทันที
          ↓
ขั้น 7: บันทึก trace ทุกการตัดสินใจพร้อมเหตุผล
```

**Risk Score มาจากอะไรบ้าง:**
- จำนวนครั้งที่ขอคืนเงินในอดีต
- มูลค่าสินค้าที่ขอคืน
- ระยะเวลาหลังซื้อ
- สถานะลูกค้า (ใหม่ / ประจำ / VIP)
- ประวัติการขอคืนเงินที่ผิดปกติ

**การอัปโหลดหลักฐาน:**
ลูกค้าแนบรูปหรือเอกสารได้ผ่านระบบ ไฟล์จะถูกเก็บใน MinIO (คล้าย S3) พร้อม link ที่ปลอดภัยสำหรับ Admin ดู

## 7. Workflow 03 — แจ้งเตือนเชิงรุก (Proactive)

**สิ่งที่ทำให้ระบบนี้พิเศษ:**  
ระบบ *รู้ก่อน* ว่าจะมีปัญหา แล้วแจ้งลูกค้าก่อนที่เขาจะต้องโทรมาถาม

**เมื่อไหร่ระบบ trigger เอง:**
- พัสดุไม่มีการอัปเดตสถานะเกิน **48 ชั่วโมง**
- พัสดุอยู่ใน "transit" นานกว่าปกติ
- Carrier ไม่ตอบสนอง

**ลำดับการทำงานอัตโนมัติ 11 ขั้น:**

```
ขั้น  1: รับ event ว่ามีพัสดุที่น่าสงสัย
ขั้น  2: ค้นหาข้อมูลพัสดุ + ออเดอร์ + ลูกค้าที่เกี่ยวข้อง
ขั้น  3: ประเมินสถานะพัสดุ: ล่าช้าแน่ไหม? ล่าช้าแค่ไหน?
ขั้น  4: ค้นหานโยบาย: พัสดุประเภทนี้ชดเชยได้แค่ไหน?
ขั้น  5: สร้าง Alert — บันทึกว่ามีพัสดุที่มีปัญหา
ขั้น  6: Supervisor ตรวจสอบ: Risk Score เท่าไหร่?
ขั้น  7: สร้าง Case — เปิดเคสสนับสนุนในระบบ Admin
ขั้น  8: อนุมัติ — ถ้า Risk ≥ 70 รอ Admin; ถ้าต่ำกว่า approve อัตโนมัติ
ขั้น  9: แจ้ง Carrier — บันทึก event ว่าได้ติดต่อบริษัทขนส่งแล้ว
ขั้น 10: บันทึกความจำ — จำไว้ว่าจัดการกับพัสดุนี้อย่างไรแล้ว
ขั้น 11: บันทึก trace ทั้งหมดให้ตรวจสอบได้
```

**สิ่งที่ลูกค้าเห็นใน portal:**
แท็บ "การแจ้งเตือน" จะแสดงข้อความเช่น:
> "พัสดุของคุณ (ORD-1001) ล่าช้ากว่าที่คาด เราได้ติดต่อ Flash Express แล้ว และจะได้รับคูปอง 50฿ เป็นการชดเชย"

**สิ่งที่ Admin เห็น:**
Pending Approval รายการใหม่ พร้อมข้อมูลครบถ้วน: ลูกค้าคือใคร, พัสดุไหน, เหตุผล, Risk Score, และ AI แนะนำให้ทำอะไร

## 8. ระบบ Memory — AI จำอะไรได้บ้าง

AI ของ ShopEasy มีความจำ **3 ระดับ** ทำงานพร้อมกัน:

### ความจำระยะสั้น (Short-term Memory)
- เก็บใน **Redis** (ความเร็วสูง)
- จำสิ่งที่เกิดขึ้นใน **session ปัจจุบัน**
- เช่น: "ลูกค้าเพิ่งถามถึง ORD-1001 เมื่อ 2 นาทีที่แล้ว"
- หายไปเมื่อ session สิ้นสุด

### ความจำระยะกลาง (Episodic Memory)
- เก็บใน **PostgreSQL**
- จำ **ประวัติการสนทนา** ในช่วง 30 วัน
- เช่น: "ลูกค้าคนนี้เคยถามเรื่องพัสดุล่าช้า 3 ครั้งในเดือนที่แล้ว"
- ช่วยให้ AI ไม่ต้องถามซ้ำ

### ความจำระยะยาว (Long-term Memory)
- เก็บใน **PostgreSQL**
- จำ **รูปแบบพฤติกรรม** ของลูกค้า
- เช่น: "ลูกค้า CUST-001 มักสั่งสินค้าต้น เดือน มีประวัติดี ไม่เคยขอคืนเงินโดยไม่มีเหตุผล"
- ใช้ปรับการตอบสนองและความเสี่ยง

**ตัวอย่างที่เห็นได้ชัด:**
ครั้งแรกที่ลูกค้าถาม AI จะถามว่า "หมายถึงออเดอร์ไหน?"  
ครั้งที่สองถ้าถามเรื่องเดิมในวันเดียวกัน AI จะรู้เลยว่าหมายถึง ORD-1001

## 9. ระบบ Approval — มนุษย์ตรวจสอบก่อน Execute

ShopEasy ใช้หลักการ **Human-in-the-Loop** ซึ่งหมายความว่า:  
AI ทำงานส่วนใหญ่เองได้ แต่การตัดสินใจที่มีผลกระทบสูงต้องผ่านมนุษย์ก่อน

### เมื่อไหร่ที่ต้องรอ Admin Approve

| สถานการณ์ | Risk Score | ต้องทำอะไร |
|-----------|-----------|-----------|
| คืนเงินปกติ มูลค่าต่ำ | < 70 | AI อนุมัติเองอัตโนมัติ |
| คืนเงินมูลค่าสูง หรือบ่อยผิดปกติ | 70–89 | Admin ต้อง review และ approve |
| กรณีพิเศษ หรือน่าสงสัยมาก | ≥ 90 | Admin ต้องดูเอกสารประกอบด้วย |

### Admin เห็นอะไรเมื่อต้อง Approve

Admin จะเห็น card ที่ประกอบด้วย:
- **ชื่อลูกค้า** และประวัติโดยย่อ
- **ออเดอร์ที่เกี่ยวข้อง** พร้อมมูลค่า
- **เหตุผลที่ AI ประเมินว่าต้องขอ Approve** เขียนเป็นภาษาไทยชัดเจน
- **Risk Score** พร้อมปัจจัยที่ทำให้คะแนนสูง
- **ปุ่ม Approve / Reject** พร้อมช่องใส่หมายเหตุ

ทุกการ Approve/Reject ถูกบันทึกว่าใครทำ เมื่อไหร่ และมีความเห็นอะไร

## 10. Observability — ดูได้ว่า AI คิดอะไรอยู่

**Observability** คือความสามารถในการ "มองเข้าไปในหัว AI" ได้

### Agent Trace — บันทึกทุก Run

ทุกครั้งที่ Workflow รัน จะมีการบันทึก Trace ที่ประกอบด้วย:
- ชื่อ Workflow ที่รัน
- เวลาเริ่มต้น – สิ้นสุด
- Nodes ที่ผ่าน ตามลำดับ
- Intent ที่ตรวจพบ
- คำตอบสุดท้ายที่ส่งให้ลูกค้า
- สถานะ: สำเร็จ / ต้องการ human / ล้มเหลว

### Tool Log — บันทึกทุก Tool Call

เมื่อ AI ใช้ tool ไหนก็จะบันทึกว่า:
- Tool อะไรที่ถูกเรียก
- ข้อมูลที่ส่งไปให้ tool
- ผลลัพธ์ที่ได้กลับมา
- ใช้เวลาเท่าไหร่

**ตัวอย่าง Tool Log:**
```
Tool: get_shipment_status
Input: shipment_id = "SHP-9001"
Output: { status: "in_transit", last_update: "2026-05-12 10:30", carrier: "Flash Express" }
Duration: 42ms
```

**ทำไมถึงสำคัญ:**  
ถ้า AI ตอบผิด หรือให้ข้อมูลผิดพลาด ทีม AI Engineer สามารถตรวจสอบได้ทันทีว่าเกิดอะไรขึ้นในแต่ละขั้นตอน เหมือน CCTV ของ AI

## 12. โครงสร้าง Services ที่รัน

ระบบทั้งหมดรันผ่าน **Docker Compose** ซึ่งมี 6 services:

### 6 Services ที่ทำงานพร้อมกัน

```
┌─────────────────────────────────────────────────┐
│                                                 │
│   [Frontend — Port 5173]                        │
│   React + TypeScript + Vite                     │
│   หน้าจอทั้ง 3 portals                           │
│                                                 │
├─────────────────────────────────────────────────┤
│                                                 │
│   [Backend — Port 8000]                         │
│   FastAPI + Python                              │
│   API, Business Logic, AI Workflows             │
│                                                 │
├──────────┬──────────┬──────────┬────────────────┤
│          │          │          │                │
│[PostgreSQL [Redis    [MinIO    [Qdrant          │
│ Port 5433] Port 6379] Port 9000] Port 6333]    │
│ ฐานข้อมูล  Cache &   เก็บไฟล์   Vector DB       │
│ หลัก       Memory    หลักฐาน   สำหรับ RAG      │
│                                                 │
└─────────────────────────────────────────────────┘
```

### ทำไมต้องมี Redis?
Redis เป็น in-memory database ที่เร็วมาก ใช้สำหรับ:
- เก็บ short-term memory ของ AI (ที่ต้องอ่านเร็วมาก)
- Session caching
- สิ่งที่ต้องการความเร็วสูงแต่ไม่ต้องเก็บถาวร

### ทำไมต้องมี MinIO?
MinIO คือที่เก็บไฟล์ (Object Storage) คล้าย Amazon S3:
- เก็บรูปภาพและเอกสารหลักฐานการคืนสินค้า
- สร้าง "presigned URL" ที่ปลอดภัย ให้ลูกค้าอัปโหลดได้โดยตรงโดยไม่ผ่าน server
- Admin สามารถเปิดดูไฟล์ได้ผ่าน link ชั่วคราวที่หมดอายุอัตโนมัติ

### ทำไมต้องมี Qdrant?
Qdrant คือ Vector Database สำหรับระบบ RAG (Retrieval-Augmented Generation):
- เก็บ embedding ของนโยบายและเอกสารทั้งหมด (ใช้โมเดล bge-m3 ขนาด 1024 มิติ)
- เมื่อลูกค้าถามคำถาม AI จะค้นหานโยบายที่เกี่ยวข้องโดยใช้ **Hybrid Search** (keyword + vector)
- ทำให้ AI ตอบคำถามเกี่ยวกับนโยบายได้อย่างถูกต้องจากเอกสารจริง ไม่ได้แต่งคำตอบเอง

ระบบมี **3 บทบาท** แต่ละบทบาทเห็นและทำได้ต่างกัน:

| บทบาท | ทำอะไรได้ | ทำอะไรไม่ได้ |
|------|---------|------------|

| AI Engineer | ดู traces ทั้งหมด, run workflow, ดู metrics | อนุมัติ/ปฏิเสธคำขอ |

### วิธี Login ที่รองรับ
1. **Email + Password** — สำหรับ demo (มี 3 account ตัวอย่าง)
2. **Google OAuth** — ใช้บัญชี Google login ได้โดยตรง

### Token-based Authentication
เมื่อ login สำเร็จ ระบบออก **JWT Token** ให้ ทุก request ที่ส่งมาจาก frontend จะต้องแนบ token นี้ไปด้วยเสมอ ถ้าไม่มีหรือหมดอายุ จะถูก reject ทันที

### Secret Management
- API Key ต่างๆ ไม่เขียนลงใน code โดยตรง
- ใช้ Environment Variable เก็บไว้ใน `.env` file
- `.env` file ถูก ignore จาก git (ไม่ push ขึ้น GitHub)

## 15. ระบบ RAG — AI ค้นหาความรู้จากเอกสาร

### RAG คืออะไร

RAG (Retrieval-Augmented Generation) คือวิธีที่ทำให้ AI ตอบคำถามจาก **เอกสารจริง** แทนที่จะแต่งคำตอบเอง

### ทำไมต้องมี RAG

ถ้าไม่มี RAG:
> ลูกค้าถาม "นโยบายคืนเงินเป็นยังไง" → AI ตอบจากความรู้ทั่วไป → **อาจผิด**

ถ้ามี RAG:
> ลูกค้าถาม "นโยบายคืนเงินเป็นยังไง" → AI ค้นหาเอกสาร Refund Policy → ดึงข้อมูลจริง → **ตอบถูกต้อง**

### ขั้นตอนการทำงาน

```
ขั้น 1: Admin อัปโหลดนโยบาย (เช่น Refund Policy, Shipping Policy)
          ↓
ขั้น 2: ระบบตัดเอกสารเป็น "ชิ้นเล็กๆ" (chunks) ~500 ตัวอักษร
          ↓
ขั้น 3: แปลงแต่ละชิ้นเป็น "ตัวเลข" (embedding) ด้วยโมเดล bge-m3
          ↓
ขั้น 4: เก็บ embedding ไว้ใน Qdrant (Vector Database)
          ↓
[เมื่อลูกค้าถามคำถาม]
ขั้น 5: แปลงคำถามเป็น embedding เช่นกัน
          ↓
ขั้น 6: ค้นหา chunks ที่ "ใกล้เคียง" กับคำถามมากที่สุด (Hybrid Search)
          ↓
ขั้น 7: ส่ง chunks ที่เจอให้ LLM พร้อมคำถาม → ตอบจากข้อมูลจริง
```

### Hybrid Search — ค้นหาสองแบบพร้อมกัน

ระบบค้นหาแบบ **ผสม** เพื่อความแม่นยำสูงสุด:

| วิธี | ทำงานยังไง | เก่งเรื่อง |
|------|-----------|-----------|
| Keyword Search | ค้นหาคำตรงๆ ใน text (ILIKE) | คำเฉพาะ เช่น "คืนเงิน", "7 วัน" |
| Vector Search | เปรียบเทียบความหมาย (embedding cosine similarity) | ความหมายคล้ายกัน เช่น "ขอเงินคืน" ≈ "refund" |

ระบบลอง Vector Search ก่อน ถ้าไม่เจอจะ fallback ไปใช้ Keyword Search

### นโยบายที่มีในระบบ

| ID | ชื่อนโยบาย | หมวด | จำนวน chunks |
|----|-----------|------|-------------|
| POL-001 | Refund Policy | refund | 6 |
| POL-002 | Return Policy | return | 5 |
| POL-003 | Compensation Policy | compensation | 5 |
| POL-004 | Shipping Policy | shipping | 6 |
| POL-005 | Seller SLA Policy | seller | 4 |

## 16. AI เข้าถึงข้อมูลลูกค้าทั้งหมด

### หลักการ: ทุกคำถามโหลดข้อมูลก่อนตอบ

ไม่ว่าลูกค้าจะถามอะไร — ถามชื่อ ถามออเดอร์ ถามนโยบาย ถามสถานะพัสดุ — AI จะ **โหลดข้อมูลทั้งหมดของลูกค้าคนนั้น** ขึ้นมาก่อนเสมอ

### ข้อมูลที่ AI เข้าถึงได้

```
[Router] → จำแนกว่าคำถามเรื่องอะไร
    ↓
[Context] → โหลดข้อมูลลูกค้าทั้งหมด:
    ├── ชื่อ, อีเมล, เบอร์โทร, ระดับสมาชิก (tier)
    ├── ออเดอร์ทั้งหมด + สินค้าในแต่ละออเดอร์
    ├── สถานะพัสดุทุกชิ้น + carrier + tracking
    └── ประวัติคำขอคืนเงินทั้งหมด
    ↓
[Memory] → โหลดความจำ 3 ชั้น:
    ├── Short-term: เพิ่งคุยเรื่องอะไรใน session นี้
    ├── Episodic: ประวัติสนทนา 30 วันย้อนหลัง
    └── Long-term: รูปแบบพฤติกรรมลูกค้า
    ↓
[ตอบ] → AI มีข้อมูลครบถ้วน ตอบได้ทุกคำถาม
```

### ตัวอย่างที่ AI ตอบได้

| คำถาม | AI รู้จาก |
|-------|----------|
| "ฉันชื่ออะไร" | ข้อมูลลูกค้า (customer.name) |
| "ฉันมีออเดอร์อะไรบ้าง" | active_orders (ทุก order + items) |
| "ของฉันส่งถึงไหนแล้ว" | active_shipments (status, carrier, ETA) |
| "ฉันเคยขอคืนเงินกี่ครั้ง" | refund_requests history |
| "นโยบายคืนเงินเป็นยังไง" | RAG → policy documents |
| "เมื่อวานฉันถามเรื่องอะไร" | Memory (episodic + short-term) |
